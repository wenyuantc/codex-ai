这个项目现在最缺的，不是再补几个页面，而是把它从“能跑的本地 demo”抬到“可演进的桌面产品”的那层骨架。最核心的判断是：你现在应该走模块化单体
（modular monolith），而不是拆服务；但这个单体里还缺 5 个关键层：应用服务层、统一数据模型、真实运行时控制、配置与观测、测试与文档。

P0 领域边界和数据边界没立住。前端直接拿 SQL 插件读写数据库 src/lib/database.ts:1，业务规则散落在各个 store 里，比如任务创建、状态变更、删除、
日志写入都在 src/stores/taskStore.ts:54，项目删除还要手工级联多张表 src/stores/projectStore.ts:57。这类结构短期快，长期会让约束、事务和迁移越
来越难控。项目缺的是一个统一的应用服务层，不管放 Rust/Tauri command 还是 TS repository/service，至少要把“UI 状态”和“业务写路径”分开。

P1 数据模型现在是自相矛盾的。Schema 同时有 employees.project_id 和 project_employees 两套归属关系 src-tauri/src/db/migrations.rs:25 src-
tauri/src/db/migrations.rs:126，但员工创建/编辑只写前者 src/components/employees/CreateEmployeeDialog.tsx:53，项目详情页却读后者 src/pages/
ProjectDetailPage.tsx:49。这会导致“关联项目”和“团队成员”天然漂移。另外 employee_metrics 表建了也有前端读取 src-tauri/src/db/migrations.rs:107
src/components/dashboard/EmployeePerformanceChart.tsx:23，但仓库里没有对应写入路径，说明分析层还是空的。

P1 配置、信任边界和可运维性也没补上。设置页里 Codex 连接状态是写死的“未连接” src/pages/SettingsPage.tsx:63，没有实际环境探测、版本检查、权限
校验。任务执行时把项目 repo_path 直接当 workingDir 传给 CLI src/components/tasks/TaskDetailDialog.tsx:102 src-tauri/src/codex/process.rs:82，
但没有路径存在性、允许范围、仓库合法性检查。项目缺少最基本的运行前自检、结构化日志、失败归因和审计策略。

P2 产品化交付层也明显不足。README 还停留在 Tauri 模板 README.md:1，说明系统边界、使用方式、数据模型、运行要求都还没被正式定义。测试方面，
cargo test --manifest-path src-tauri/Cargo.toml 能过，但结果是 0 tests；前端也没有自动化测试。也就是说你现在缺的不是“更多功能覆盖”，而是“变更
安全网”。

建议路线

主推荐是继续保持桌面单体，不要拆微服务，但尽快补齐 4 个模块：project/task/employee 的应用服务层，codex_runtime 的会话与进程层，analytics 的指
标写入层，settings/health 的配置与环境探测层。前端 store 只负责缓存和交互，不直接拼 SQL。

如果只排一个最小路线，我会这样做：

1. 先修运行时模型：加 codex_sessions/事件日志表，理顺 start/stop/restart/status，明确是否支持交互式 stdin。
2. 再统一团队归属模型：二选一，只保留 employees.project_id 或 project_employees，不要双轨。
3. 然后收口数据写路径：把任务、项目、员工的写操作收进 Tauri commands/service，前端停止直接执行 SQL。
4. 最后补交付底座：README、ADR、最小测试集、设置页健康检查。

不建议现在做的事是拆后端服务、上远程同步、搞复杂 agent 编排平台。当前规模下，那会把复杂度加在错误的位置。
