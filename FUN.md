# FUN

## 说明

- 本文基于 2026-04-14 对当前仓库代码的盘点整理，主要参考 `README.md`、`src/App.tsx`、`src/pages/*`、`src/components/*`、`src/stores/*`、`src-tauri/src/app.rs`、`src-tauri/src/codex/*`、`src-tauri/src/db/*`。
- 判断原则：只把“当前代码里已经形成页面入口、交互入口或完整后端能力”的内容算作现有功能；对“已有数据结构/已有组件但闭环不完整”的能力，会单独列到“已有雏形但未闭环”。

## 产品判断

- 当前项目已经不是一个普通的任务管理器，而是一个“围绕本地 Git 仓库任务执行的 AI 协作桌面工作台”。
- 当前最强的能力是：任务执行、AI 员工调度、Codex 会话管理、代码审核、任务级图片上下文、运行日志与本地数据掌控。
- 当前最弱的能力是：项目管理闭环、交付管理闭环、全局搜索与通知、会话中心、Git 工作流可视化、团队运营视角。
- 换句话说：现在更像“执行中台”，还不够像“完整产品管理台”。

## 现有功能清单

### 1. 项目管理

- 新建项目。
- 编辑项目名称、描述、状态。
- 删除项目。
- 将项目标记为活跃或归档。
- 为项目配置本地仓库路径。
- 新建项目时可根据选择的目录自动回填项目名。
- 在顶部切换“当前项目”，让仪表盘、看板、员工等页面按项目范围过滤。
- 项目列表支持按全部 / 活跃 / 归档筛选。
- 已有项目详情页，可展示项目描述、任务状态统计、任务列表、团队成员。

### 2. 任务管理

- 看板页展示五种任务状态：`todo`、`in_progress`、`review`、`completed`、`blocked`。
- 支持在看板中拖拽任务跨列流转状态。
- 新建任务。
- 任务支持标题、描述、优先级、所属项目、指派员工。
- 创建任务时支持选择图片附件。
- 任务图片会复制到应用托管目录，不依赖原始路径持续存在。
- 任务详情支持修改标题、描述、状态、优先级、负责人、审查员。
- 删除任务。
- 子任务增删改状态。
- 子任务支持去重插入，避免 AI 或人工重复拆分。
- 任务评论区支持人工评论。
- 任务评论区支持 AI 生成评论并落库。
- 任务详情支持图片附件预览、打开、删除、补充上传。
- 任务级运行日志弹窗可查看执行输出。
- 支持基于上一次执行会话继续对话。
- 任务会记录最近一次执行会话 ID。
- 任务会记录最近一次审核会话 ID。
- 任务详情可查看 execution 会话的文件变更历史。
- 任务支持复杂度字段，并可在卡片上展示复杂度结果。
- 任务支持 AI 建议文本持久化保存。

### 3. AI 员工管理

- 新建 AI 员工。
- 编辑 AI 员工。
- 删除 AI 员工。
- 员工支持角色：开发者、审查员、测试员、协调员。
- 员工支持模型配置。
- 员工支持推理强度配置。
- 员工支持 specialization。
- 员工支持 system prompt。
- 员工支持归属到某个项目。
- 员工列表支持按状态筛选：全部 / 在线 / 忙碌 / 离线 / 错误。
- 员工状态会结合运行时会话自动推导，不只是静态字段。
- 员工卡可直接启动 Codex。
- 员工卡可直接停止 Codex。
- 员工卡可直接重启 Codex，并输入新的任务描述。
- 员工卡自带终端日志折叠区。

### 4. Codex 执行与会话

- 任务可直接运行 Codex。
- 任务可直接停止 Codex。
- 员工可从符合条件的任务列表中选择任务后启动 Codex。
- 任务执行 prompt 会自动拼接标题、描述、子任务、图片附件。
- 续聊时会基于历史 session 继续发送新的补充指令。
- 任务图片会在启动和续聊时自动附带给 Codex。
- 支持 execution 和 review 两类会话。
- 会话会写入本地数据库，记录状态、工作目录、退出码、CLI session id、resume session id。
- 会话事件会落库，保留失败原因、审核报告等。
- 运行过程中支持实时 stdout/stderr 事件流。
- 任务级日志和员工级日志都能实时查看。
- 支持记录 execution 会话前后 Git 工作区文件变化，并沉淀为结构化文件变更历史。

### 5. 代码审核

- 任务在 `review` 状态下可发起代码审核。
- 代码审核必须指定审查员。
- 代码审核会基于项目仓库路径收集当前工作区改动上下文。
- 审核上下文会包含 diff 与部分未跟踪文件片段。
- 审核会话日志可在任务详情里查看。
- 最近一次审核结果可在任务详情里查看。
- 审核结果支持复制。
- 审核结果支持一键创建“修复任务”。
- 修复任务可直接继承审核结果并立即交给原开发负责人执行。

### 6. 一次性 AI 能力

- AI 建议指派。
- AI 分析复杂度。
- AI 拆分子任务。
- AI 生成任务计划。
- AI 生成评论。
- AI 生成计划后可预览，并插入任务详情。
- 一次性 AI 操作会输出执行日志，显示当前模型、推理强度、执行通道。
- 一次性 AI 支持带图片上下文执行。

### 7. 仪表盘与分析

- 仪表盘显示活跃项目数。
- 仪表盘显示总任务数。
- 仪表盘显示在线员工数。
- 仪表盘显示完成率。
- 仪表盘显示任务状态分布。
- 仪表盘显示最近活动。
- 最近活动支持手动刷新。
- 最近活动支持 30 秒自动刷新。
- 最近活动支持查看更多弹窗，并带分页。
- 仪表盘显示员工绩效表。
- 员工绩效支持近 7 / 30 / 90 天切换。
- 员工绩效展示完成任务数、平均耗时、成功率。
- 任务完成时会写入员工绩效指标。

### 8. 设置、数据与运维

- 设置页支持亮色 / 暗色 / 跟随系统三种主题模式。
- 头部也支持快速明暗切换。
- 设置页可检查 Codex CLI 是否可用。
- 设置页可显示 `codex --version`。
- 设置页可检查 Node 是否可用。
- 设置页可检查 Codex SDK 是否已安装。
- 设置页可显示数据库路径、迁移版本、最近一次迁移说明。
- 设置页可显示最近一次会话失败原因。
- 设置页支持开启 / 关闭任务执行走 SDK。
- 设置页支持开启 / 关闭一次性 AI 走 SDK。
- 设置页支持配置一次性 AI 默认模型。
- 设置页支持配置一次性 AI 默认推理强度。
- 设置页支持 Node 路径覆盖。
- 设置页支持安装或重装 Codex SDK。
- 设置页支持打开数据库文件夹。
- 设置页支持导出 SQL 备份。
- 设置页支持导入 SQL 备份。
- 导入 SQL 之前会自动备份当前数据库。
- 应用内展示键盘快捷键说明。

### 9. 数据与架构层能力

- 本地 SQLite 持久化。
- 数据库迁移版本化管理。
- 活动日志持久化。
- 员工绩效持久化。
- Codex 会话与事件持久化。
- 任务附件独立托管存储。
- 审核结果以事件形式保存。
- 文件变更历史以结构化表保存。
- 前端使用 Zustand 管理跨页面状态。
- 后端已把核心写路径统一收拢到 Tauri commands。

## 已有雏形但未闭环

- 项目详情页已经存在，但当前项目列表和其他主页面没有直接跳转入口，属于“有页面、弱入口”。
- `shortcut:new-task`、`shortcut:toggle-employees`、`shortcut:command-palette` 事件已经发出，但当前未看到接收端，说明快捷能力还没真正闭环。
- 员工卡上已有“工作负载”进度条，但员工列表没有传入真实任务数，当前更像 UI 占位而非有效指标。
- execution 文件变更历史已经很强，但只停留在任务详情视角，没有独立的“会话中心 / 改动中心”。
- 审核结果已经能派生修复任务，但还没有“审核通过 / 审核驳回 / 已修复回流原任务”的完整状态闭环。
- 主题能力有 Header 快捷切换和 Settings 配置两套入口，但还没有统一的全局主题状态管理体验。

## 从产品视角还缺少什么功能

### P0：最应该优先补齐

- 全局搜索：现在几乎没有针对项目、任务、员工、会话的全文搜索能力。
- 交付时间管理：缺少截止日期、里程碑、迭代、计划开始时间，导致“交付承诺”不可管理。
- 依赖与阻塞原因：虽然任务有 `blocked` 状态，但没有阻塞原因、解除条件、依赖关系。
- 通知中心：缺少“审核待处理、运行失败、任务完成、SDK 不可用、数据库异常”等主动提醒。
- 全局会话中心：缺少按项目 / 员工 / 任务查看历史执行、失败、恢复、review 的统一视图。
- 仓库接入向导：项目配置 repo_path 后，还缺少 `.git`、当前分支、脏工作区、权限可用性等可视化预检。

### P1：第二优先级

- Git 工作流视图：缺少分支信息、改动摘要、最近提交、可回看 diff 的产品化入口。
- 任务排序与排程：当前看板支持跨列流转，但不支持列内排序、手动排优先级、批量调整。
- 丰富附件类型：现在基本聚焦图片，缺少文档、日志、PDF、设计稿等非图片上下文。
- 批量操作：缺少批量指派、批量改状态、批量归档、批量删除。
- 报表导出：活动、绩效、任务状态缺少导出或周报视图。
- 项目模板 / Prompt 模板：缺少不同项目类型、角色类型的模板化配置。

### P2：中长期再考虑

- 多工作区 / 多数据库切换。
- 团队协作角色与权限。
- 更多外部接入，如 GitHub Issue、PR、Commit 联动。
- 自动备份计划与恢复点管理。

## 现有功能还需要完善什么

### 1. 项目管理需要完善

- 项目列表卡片信息偏轻，缺少任务数、在线员工数、最近运行状态、最近活动摘要。
- 项目详情页需要真正接进主导航，而不是停留在隐藏能力。
- 仓库路径虽然可配置，但最好在项目层就明确提示“这是不是有效 Git 仓库、当前分支是什么、是否存在未提交改动”。

### 2. 看板与任务流需要完善

- 看板现在本质上是“改状态”，不是“排顺序”；建议补齐列内排序。
- `blocked` 状态需要强制填写阻塞原因，否则状态价值不大。
- 缺少截止日期、里程碑、标签、估时、实际耗时，任务管理深度还不够。
- 当前 AI 按钮很多，但任务执行后的产出沉淀还不够结构化，建议把“计划、总结、结论、审核要点”形成固定信息块。

### 3. AI 员工体系需要完善

- 员工角色已有，但缺少能力画像、默认行为模板、擅长领域、负载上限等更强的调度信息。
- 工作负载条当前未接真实数据，建议直接接入“当前进行中任务数 / 最近 24h 会话数 / 失败率”。
- 绩效指标现在偏轻，建议增加“审核通过率、平均修复轮次、最近失败数、恢复成功率”等更贴近 AI 员工的指标。

### 4. 会话与审计需要完善

- 当前会话日志较强，但缺少统一的会话检索、筛选、对比、恢复入口。
- 审核结果和 execution 文件变更历史已经可用，建议再补“从一次执行到一次审核到一次修复”的完整链路视图。
- 失败诊断已记录最近错误，但应该进一步做成可操作的修复建议，而不只是展示文本。

### 5. 设置与体验需要完善

- 主题切换入口有重复，建议合并为一个统一体验。
- 快捷键说明和实际能力存在不一致，建议只保留真实可用的快捷键，或者把事件真正接上。
- 设置页信息量较大，建议拆成“运行环境 / AI 设置 / 数据安全 / 外观偏好”四块。

## 我作为产品的建议路线

### 第一阶段：把“执行中台”补成“交付闭环”

- 做全局搜索。
- 做会话中心。
- 做通知中心。
- 做截止日期 / 里程碑 / 阻塞原因。
- 做项目详情真实入口。

### 第二阶段：把“AI 运行”补成“Git 交付”

- 做仓库预检。
- 做分支 / 改动 / commit 视图。
- 做审核通过 / 驳回 / 修复回流闭环。
- 做 execution / review / fix 的链路可视化。

### 第三阶段：把“单机工具”补成“运营后台”

- 做负载视图。
- 做更可信的绩效体系。
- 做报表与导出。
- 做模板化角色与项目配置。

## 一句话结论

- 这个项目当前最有价值的地方，不是“又一个任务管理器”，而是“把本地仓库任务、AI 员工、Codex 会话、审核修复链路收在一个桌面工作台里”。
- 现在最缺的，不是再加几个 AI 按钮，而是把“搜索、通知、时间、会话中心、Git 闭环、项目入口”补齐，这样它才会从“好玩的执行工具”真正变成“可长期使用的产品”。
